# ChatBI Domain Spec

## Intent

ChatBI should help procurement users ask business questions and receive trustworthy, traceable answers from approved data sources.

## Initial User Group

The initial users are procurement leaders and team leads. The first version should support decision review and management visibility before broad self-service analytics.

## Target Product Pattern

ChatBI should be similar in spirit to Tencent-style ChatBI products, but adapted for internal company compliance and local data control.

The backend should run locally or inside the company-approved environment. The database will be built later from SAP exports converted into database tables.

## AI Boundary

The LLM may use a company-provided internal API. Because this API is internally provided, the user's natural language question may be sent to the LLM as written, including business entities such as supplier names, project numbers, purchase order numbers, or prices when the user includes them.

The LLM must not receive:

- Real database rows.
- Raw query result details.
- Database credentials.
- Free-form SQL execution capability.
- Permission to write SQL directly.

The LLM may receive:

- The user's natural language question.
- Table headers exposed by the system.
- Hard-coded field meanings from the field dictionary.
- Tool schemas for approved BI tools.
- Small controlled metadata needed to choose a tool.

## Query Orchestration

The LLM should parse the user's natural language request into structured JSON. The JSON should describe the interpreted business intent, required dimensions, measures, filters, time range, and selected BI tool.

The backend validates the JSON before execution. The LLM does not directly query the database. Instead, it calls prebuilt BI tools that represent common procurement BI needs.

Each BI tool owns its query logic. SQL may exist inside trusted backend code, views, stored procedures, or query builders, but it is not generated freely by the LLM.

## Field Dictionary

The system should hard-code the business meaning of table headers in a field dictionary. This dictionary is used for:

- LLM intent parsing.
- Tool parameter validation.
- Result labeling.
- Frontend tooltips.
- Data source traceability.

For every exposed field, the dictionary should eventually include:

- Field name.
- Business meaning in natural language.
- Source system or source export.
- Table or view location after the database exists.
- Data type or expected format.
- Whether the field is allowed in ChatBI answers.

## Result Experience

Every ChatBI response should start with one sentence explaining how the system understood the user's natural language request.

The response should then provide structured BI results, such as tables, cards, ranked lists, trend summaries, or charts after the frontend design is approved.

Fields shown in the result should support interactive tooltips. A tooltip should explain:

- What the field means in business language.
- Which data source it comes from.
- Any relevant caveat from the field dictionary.

## Current Decision Boundary

ChatBI implementation is paused until the domain is specified through SDD conversations. The next ChatBI conversations should define:

- Priority questions users want to ask.
- Metrics and definitions users already trust.
- Source systems, files, PowerBI datasets, or reports involved.
- Required answer format.
- Permission and sensitivity boundaries.
- Cases where the assistant should refuse, ask for clarification, or show uncertainty.
- The first approved BI tools and their JSON schemas.
- The first field dictionary entries derived from SAP exports or existing BI reports.

## Candidate Question Categories

These are candidate categories, not approved requirements:

- Supplier performance.
- Purchase order status.
- Cost or price trend analysis.
- Project purchasing overview.
- Delivery or quality risk.
- Data quality checks.

## Traceability Requirement

ChatBI answers should eventually explain where the answer came from. At minimum, future specs should decide whether answers cite:

- Database table or view names.
- Report names.
- Source file names.
- Query filters.
- Time range.
- Data refresh time.

Traceability should be generated by local backend metadata and the field dictionary, not by asking the LLM to infer data lineage from raw data.

## Development Data Rule

Development on a personal computer uses dummy data. Dummy data may include synthetic supplier names, project numbers, purchase order numbers, and prices for testing the full ChatBI flow.

Real company data should be introduced only in the company-approved environment and only through approved local database or internal API paths.
